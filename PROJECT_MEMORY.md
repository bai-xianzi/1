## 0. 文件职责、权威边界与冲突优先级

为避免长期记忆不断追加后发生内容漂移，项目文件职责固定如下：

1. `PROJECT_MEMORY.md`：长期目标、硬性原则和稳定设计判断；
2. `SYSTEM_ARCHITECTURE.md`：模块层级、横向能力和依赖关系；
3. `PROJECT_STATUS.md`：当前阶段、未决问题和下一步；
4. `DEVELOPMENT_GUIDANCE.md`：开发过程必须遵守的方法；
5. `CAUSAL_MARKET_SIMULATION_ENGINE.md`：CWMS专项设计；
6. `INSTITUTIONAL_RESEARCH_AND_EXECUTION_GOVERNANCE.md`：机构化研究生产线与券商级执行安全专项原则；
7. `APPLIED_STATISTICAL_EVIDENCE_INTELLIGENCE.md`：全网应用统计证据、统计主张、反证、因果等级和候选特征治理专项原则；
8. `schemas/`：机器可读字段、枚举、变量、规则和场景契约。

发生冲突时，先停止开发并统一文件。对于“现在做到哪里”，以 `PROJECT_STATUS.md` 为准；对于模块归属，以 `SYSTEM_ARCHITECTURE.md` 为准；对于字段和枚举，以 `schemas/` 为准。

长期文件可以保留历史背景，但不得用旧的“当前状态”覆盖 `PROJECT_STATUS.md`。

## 1. 项目定位

本项目面向中国A股市场，目标是建设一套个人可长期迭代的量化投资辅助操作系统。

它不是简单的K线软件，也不是Qlib、TradingView、vn.py、聚宽、MT5等产品的拼接。平台应当吸收不同优秀项目和产品各自最成熟的能力，并围绕A股市场形成一条统一决策链：

```text
数据
→ 市场与现实逻辑
→ 参与者行为与资金状态
→ 市场风险
→ 建议仓位
→ 策略选择
→ 行业筛选
→ 个股排名
→ 目标组合
→ T+1集合竞价确认
→ 人工交易
→ 结果反馈
```

当前阶段不追求自动下单，不追求一次性建设完整机构系统。

近期最高目标是：

> 基于DolphinDB中的A股数据，每日或每周稳定产出一份可供人工执行的交易决策报告。

## 2. 甲方、项目经理与开发工具的职责

### 2.1 项目所有者（甲方）

项目所有者负责：

- 提出业务目标、使用场景、硬性约束和优先级；
- 提供能够获得的数据和现实操作条件；
- 描述希望系统解决的问题；
- 审阅项目经理提出的方案；
- 决定是否接受重大功能、风险规则和实盘使用结果；
- 对最终人工交易承担决策责任。

甲方的自然语言例子属于需求说明，不自动等同于字段、数据库表或代码实现。

### 2.2 ChatGPT（项目经理兼解决方案架构师）

项目经理负责：

- 从大局观审视需求，不机械照搬甲方举例；
- 发现需求中的歧义、冲突、过度设计和潜在逻辑漏洞；
- 将自然语言需求抽象成领域对象、字段、枚举、关系、接口和流程；
- 确定变量命名、模块边界、开发顺序和技术实现方案；
- 说明不同方案的取舍、风险和未来扩展影响；
- 维护项目长期记忆、总体结构图、字段字典和开发规则；
- 为Codex生成边界清晰的任务包；
- 验收Codex产出是否符合金融语义和总体架构。

项目经理不能因为甲方提出某个例子，就直接创建同名字段或模块；必须先进行领域抽象并向甲方说明。

### 2.3 Codex、Trae和VSCode

- Codex：在明确任务包内执行代码、测试、映射和报告；
- Trae：辅助解释、复核和局部修改；
- VSCode：主要代码工作区、运行环境和Git管理入口。

这些工具不负责决定总体架构、金融语义、字段命名和开发优先级。

### 2.4 沟通方式

- 使用中文解释；
- 一次推进一个可验收任务；
- 修改前说明目的和影响；
- 每个文件说明职责；
- 每一步提供输入、输出、运行方式和验收标准；
- 不用“代码已经写了”代替真实验证；
- 重大设计由项目经理提出专业方案和取舍，交甲方确认。

- 表：`stock_daily_k`

主要字段包括：

- 股票代码、交易日期；
- 开盘、最高、最低、收盘；
- 成交量、成交额；
- 换手率、涨跌幅；
- 多周期收益；
- 均线；
- MACD、KDJ、RSI、BIAS、BOLL；
- 流通股本、总股本；
- 复权因子、复权价格；
- 分红、送股、转增、配股等公司行为字段。

### 4.2 基本面数据

拥有简化个股基本面快照。

历史上导入曾出现：

```text
KeyError: 'source_file'
```

因此需要以DolphinDB当前实际状态为准重新验收。

### 4.3 七类行情、竞价和资金流数据

主要包括：

- `hq`：个股收盘行情；
- `hy`：行业收盘行情；
- `gn`：概念收盘行情；
- `kphq`：个股集合竞价；
- `kphy`：行业集合竞价；
- `kpgn`：概念集合竞价；
- `zj`：个股资金流。

这些文件主要覆盖2025—2026年。

### 4.4 当前没有或暂不优先使用

- 分钟K；
- L2逐笔成交；
- 逐笔委托；
- 十档盘口；
- 实时资金；
- 自动交易接口。

## 6. 项目核心决策逻辑

系统必须坚持以下先后顺序：

```text
先判断市场能不能做
→ 再决定做多少
→ 再决定使用什么策略
→ 再筛选行业和股票
→ 最后才决定买什么
```

任何模型分数都不能绕过风险和仓位模块直接产生交易指令。

系统每天应依次回答：

1. 数据是否完整、可信？
2. 市场处于牛市、震荡还是熊市？
3. 风险等级如何？
4. 建议总仓位和现金比例是多少？
5. 当前允许什么策略？
6. 哪些行业和概念值得关注？
7. 哪些股票排名最靠前？
8. 当前持仓应该保留、减仓、卖出还是调仓？
9. T+1集合竞价是否支持原判断？
10. 最终买什么、卖什么、持有什么？

## 8. 核心模块

### 8.1 数据质量模块

职责：

- 检查日期；
- 检查覆盖率；
- 检查重复；
- 检查空值；
- 检查异常值；
- 统一代码；
- 统一字段和单位；
- 检查数据是否更新；
- 关键数据异常时停止正式报告。

### 8.2 市场状态模块

第一版采用可解释规则。

可使用：

- 指数趋势；
- 市场宽度；
- 上涨和下跌家数；
- 涨停和跌停家数；
- 成交额；
- 波动率；
- 新高和新低；
- 行业扩散；
- 赚钱效应；
- 资金流。

输出：

- 牛市概率；
- 震荡概率；
- 熊市概率；
- 风险等级；
- 状态说明。

### 8.3 风险与仓位模块

输出：

- 建议总仓位；
- 建议现金比例；
- 是否允许新增仓位；
- 单股最大权重；
- 行业最大权重；
- 当前允许的策略。

上游风险拥有否决权。

### 8.4 因子模块

第一版只做约20个可解释因子。

技术和趋势：

- 5日、20日、60日收益；
- 均线趋势；
- 波动率；
- 最大回撤；
- 成交量变化；
- 换手率；
- 价格位置。

基本面：

- 市盈率；
- 市净率；
- ROE；
- 营收增长；
- 利润增长；
- 负债水平。

资金和市场：

- 个股资金流；
- 行业强度；
- 概念强度；
- 集合竞价强弱；
- 板块一致性。

### 8.5 模型模块

第一阶段只保留：

- 规则综合评分；
- 线性或逻辑回归基线；
- LightGBM主模型。

暂不优先：

- Transformer；
- 图神经网络；
- 强化学习；
- 大型模型集成。

### 8.6 股票池与排名模块

过滤：

- ST；
- 停牌；
- 上市时间过短；
- 流动性不足；
- 一字板；
- 数据缺失；
- 不符合当前风险状态的行业和股票。

综合：

```text
基本面
+ 技术面
+ 资金流
+ 行业强度
+ 概念强度
+ 模型分数
+ 竞价确认
```

### 8.7 组合模块

负责：

- 根据总仓位分配资金；
- 控制单股权重；
- 控制行业集中度；
- 限制换手；
- 对比当前和目标持仓；
- 生成买入、卖出、持有和调仓建议。

只有组合层可以输出最终目标权重。

### 8.8 真实A股回测模块

至少处理：

- T+1；
- 停牌；
- 涨停买不进；
- 跌停卖不出；
- 一字板；
- 100股整数手；
- ST和新股；
- 手续费；
- 印花税；
- 过户费；
- 滑点；
- 分红送转；
- 真实信号和成交时点。

评价：

- 年化收益；
- 最大回撤；
- 波动率；
- 夏普；
- 换手；
- 超额收益；
- 牛/震/熊分段；
- 样本外。

### 8.9 每日决策模块

T日收盘：

```text
更新数据
→ 检查数据
→ 市场状态
→ 风险仓位
→ 策略
→ 行业和股票池
→ 因子和模型
→ 目标组合
→ 日报
```

T+1竞价：

```text
读取个股、行业、概念竞价
→ 判断高开、低开、板块一致性
→ 升级、降级、剔除、重新排名
→ 最终人工交易清单
```

## 12. 工程原则

### 12.1 简化原则

```text
逻辑可以机构级
实现必须个人级
```

优先级：

```text
正确性
> 可用性
> 稳定性
> 速度
> 界面
> 复杂架构
```

### 12.2 保留

- Git管理代码；
- 一个主要配置文件；
- 中文日志；
- 按日期保存运行结果；
- 关键模块最小测试；
- 错误时停止正式清单；
- 模块化单体。

### 12.3 当前不做

- 微服务；
- Kubernetes；
- Kafka；
- Dagster/Airflow/Temporal；
- 企业级权限；
- 复杂模型注册；
- 复杂版本平台；
- 完整TradingView前端；
- 高频；
- Level-2；
- 强化学习；
- 大型深度模型；
- 自动下单。

### 12.4 失败保守

以下情况不生成正式交易清单：

- 数据未更新；
- 关键字段缺失；
- 复权失败；
- 股票覆盖率异常；
- 模型输入异常；
- 回测规则异常；
- 程序运行失败。

## 渐进式开发与避免推倒重来的硬性约束

这是本项目后续设计、开发、重构和扩展时必须长期遵守的硬性原则。

### 核心要求

```text
先建立稳定骨架
→ 当前功能先跑通
→ 后续通过新增、扩展和替换逐步增强
→ 尽量避免整体推倒重来
```

本项目允许多次二次开发和持续迭代，但每次开发都应尽量建立在现有可运行成果之上。

### 必须遵守的设计原则

1. **先有大局观，再写局部代码**
   - 开发任何模块前，先确认它位于总体结构图的哪一层；
   - 明确输入、输出、依赖和未来扩展方向；
   - 防止局部实现破坏整体架构。

2. **当前不开发的能力，也要预留合理扩展点**
   - 可以暂不实现新闻、舆情、战争、评论区、自动交易等功能；
   - 但标准对象、字段命名、接口边界和模块归属应提前考虑；
   - 预留扩展余地不等于提前创建大量空代码。

3. **稳定核心，扩展外围**
   - 核心对象、字段语义、风险主链和模块边界应尽量稳定；
   - 新功能优先通过新增模块、扩展字段、插件或内部替换加入；
   - 不轻易修改已经被多个模块共同依赖的核心定义。

4. **兼容已有数据和已有功能**
   - 新字段优先新增，不随意改名或删除；
   - 需要修改时提供映射、兼容或迁移方案；
   - 新版本不能无理由破坏已有脚本、数据库、报告和回测结果。

5. **先跑通，再增强**
   - 第一版允许功能简单，但必须可验证、可运行；
   - 后续通过小步迭代替换内部实现；
   - 不因追求完美架构而长期无法产生可用结果。

6. **避免提前过度设计**
   - 不为遥远功能一次性建设复杂微服务、权限或分布式系统；
   - 只设计稳定边界和必要扩展点；
   - 实际功能在需要时再实现。

7. **每次重大改动必须做影响分析**
   - 会影响哪些模块；
   - 是否影响字段字典；
   - 是否影响DolphinDB数据；
   - 是否影响回测口径；
   - 是否影响历史结果；
   - 是否需要数据迁移或兼容层。

8. **优先采用可替换实现**
   - 对外暴露稳定接口；
   - 内部实现可以从规则版升级为模型版；
   - 可以从简单报告升级为Web工作台；
   - 可以从人工交易升级为自动交易，而不重写上游决策链。

### 应尽量稳定

- 统一金融语义；
- 标准字段字典；
- 核心领域对象；
- 市场状态—风险—仓位—策略—组合主链；
- DolphinDB作为正式数据底座；
- 模块之间的输入输出边界；
- 数据时间、复权和未来数据防护原则。

### 可以逐步替换或增强

- 因子数量；
- 模型算法；
- 市场状态计算方法；
- 风险阈值；
- 组合优化方法；
- 回测细节；
- 报表形式；
- 前端界面；
- 券商接口。

### 每次新增功能的必答问题

```text
1. 它属于总体结构图哪一层？
2. 当前为什么需要？
3. 当前最小实现是什么？
4. 未来可能扩展成什么？
5. 现在需要预留什么边界？
6. 会不会破坏已有数据和功能？
7. 能否通过新增而不是重写实现？
8. 如何验证旧功能仍然可用？
```

### 总结原则

```text
有长期架构
但不过度建设

有扩展余地
但不堆空模块

允许多次二次开发
但尽量不推倒重来

当前功能先跑通
未来能力逐步增强
```

## 通用多源数据架构与适配原则

本项目不能围绕某一家厂商、某一个数据库或当前某张表反向设计整个系统。

正确的大局观是：

```text
多种外部和自定义数据源
→ 原始数据保留
→ 数据源适配与领域语义适配
→ 统一字段字典和标准对象
→ 数据质量、血缘与可用时间校验
→ 标准数据服务
→ 市场状态、风险、因子、模型、组合、回测和报告
```

### 数据源范围

系统长期必须能够容纳：

- 上交所、深交所、北交所及监管机构公开数据；
- Wind、同花顺、东方财富、聚宽等厂商数据；
- Tushare、AkShare及其他公开API；
- DolphinDB、MySQL、PostgreSQL等数据库；
- Excel、CSV、JSON、Parquet等本地文件；
- 用户自定义数据源；
- 新闻、公告、舆情、评论区、专家观点；
- 战争、制裁、地缘冲突和实体经济等远期数据。

### 字段字典的定位

`canonical_fields.yaml` 是平台内部金融语义的最终标准。

外部来源可以采用不同：

- 编程语言；
- 字段名称；
- 股票代码格式；
- 数据单位；
- 时间格式；
- 状态码；
- 缺失值规则；
- 复权口径。

但进入平台标准层以后，同一个金融概念必须只有一套明确、可追溯、可验证的标准含义。

### 适配器的准确含义

这里的适配器是数据源适配器和语义适配器，不是把各类软件整体连接在一起。

适配器负责：

- 发现来源表结构和字段；
- 提取原始数据；
- 映射到标准字段；
- 统一证券代码和交易所；
- 转换单位、比例和数据类型；
- 处理日期、时区、报告期、公告期和可用时间；
- 转换复权口径和公司行为语义；
- 映射枚举、状态码和缺失值；
- 执行质量校验；
- 保存来源、映射版本和转换血缘；
- 输出标准对象或标准数据表。

### 原始层、标准层和业务层

必须保留分层：

```text
Raw Layer
来源原始数据、原始字段、原始状态和获取时间
        ↓
Adapter / Mapping Layer
字段映射、代码统一、单位转换、时间和枚举转换
        ↓
Canonical Layer
统一字段和标准领域对象
        ↓
Quality / Lineage Layer
质量报告、来源追溯、映射版本、可用时间和冲突记录
        ↓
Semantic / Feature Layer
市场状态、因子、标签、模型输入和业务视图
```

不能导入后直接改列名并丢失原始语义。

### 适配器按来源和领域共同拆分

不建设一个无限膨胀的“万能适配器”。

推荐结构：

```text
adapters/
├─ dolphindb/
│  ├─ daily_bar_adapter.py
│  ├─ fundamental_adapter.py
│  └─ auction_adapter.py
├─ wind/
│  ├─ daily_bar_adapter.py
│  ├─ fundamental_adapter.py
│  └─ industry_adapter.py
├─ exchange/
│  ├─ sse_instrument_adapter.py
│  └─ szse_disclosure_adapter.py
└─ custom/
   ├─ excel_adapter.py
   ├─ sentiment_adapter.py
   └─ geopolitical_event_adapter.py
```

### 多来源冲突与来源角色

同一标准字段可以来自多个来源，并预留：

- 主来源；
- 备用来源；
- 对账来源；
- 来源优先级；
- 冲突记录；
- 人工确认状态。

当前不要求立即完成自动切换，但架构必须允许以后增加。

### DolphinDB的准确定位

DolphinDB是当前阶段的正式数据底座和当前运行事实源，但不是系统架构的唯一边界。

当前DolphinDB数据应作为通用数据接入骨架的第一个真实适配案例：

```text
先定义通用契约
→ 再实现DolphinDB适配器
→ 再用现有日K、基本面和快照数据验收通用链路
```

禁止为了当前表结构方便，把来源字段直接写入市场状态、因子、模型和回测模块。

### 稳定的数据契约

以下内容应尽量稳定：

- 标准字段字典；
- 标准领域对象；
- 数据源适配器接口；
- 来源映射文件格式；
- 数据质量结果格式；
- 数据血缘格式；
- 时间与可见性原则；
- 来源冲突和确认状态；
- 标准数据服务对业务模块的输入输出。

以后增加新厂商或自定义数据源，应优先新增适配器和映射文件，不修改下游业务模块。

### 数据架构正式开发顺序

```text
1. 固化项目权威文件和统一字段字典
2. 定义通用数据源接口
3. 定义Raw、Canonical、Semantic分层
4. 定义来源映射规范
5. 定义质量报告和血缘规范
6. 实现DolphinDB日K首个适配器
7. 用真实样例验证通用链路
8. 继续实现基本面和七类快照适配器
9. 再进入市场状态与风险仓位MVP
```

## 因果驱动的世界—经济—行为—资本—市场仿真引擎

本项目正式纳入 `Causal World-to-Market Simulation Engine (CWMS)`。

它不是随机K线工具，而是横向服务行情预测、因子挖掘、多因子选股、策略回测、组合、风险仓位和整套系统测试的共同研究验证底座。

### 核心链路

```text
标准字典中的世界变量与通用事件
→ 真实/人工/场景/代理/专家/估计/推导/合成方式激活
→ 因果规则和专家硬逻辑
→ 企业行业暴露
→ 企业经营状态
→ 参与者行为
→ 资本在股票、房地产、存款、理财、债券、黄金等资产间迁移
→ 市场牛熊、流动性和行业结构
→ 具体日K、分时、逐笔和盘口路径
```

### 硬性原则

- 大体牛熊、流动性和行业结构不得直接随机；
- 随机性只用于参数范围、反应时滞和具体市场路径；
- 结构变量不得任意随机；
- 内生变量优先由上游变量和因果规则推导；
- 事件不能直接映射成股票涨跌，必须经过企业经营、参与者行为和资金迁移；
- 硬恒等、硬约束、物理关系、会计关系和市场制度不得违反；
- 条件因果必须记录适用条件、失效条件、时滞、强度和证据；
- 真实、人工、专家、估计、推导和合成值必须明确区分；
- 每条合成行情必须可解释、可追溯和可复现；
- 合成行情使用统一标准字段，业务模块不得为合成数据建立另一套语义。

### 标准字典升级

标准字典不再只是等待真实数据源填充的被动目录，而是仿真引擎的变量宇宙。

所有市场相关变量必须拥有至少一种可激活方式：

- 真实观测；
- 报告披露；
- 人工输入；
- 场景假设；
- 专家评估；
- 代理估计；
- 因果推导；
- 隐状态估计；
- 合理分布采样；
- 合成生成。

### 用户可操纵变量与场景控制平面

CWMS必须提供“用户场景控制平面”。标准字典中凡是可能影响市场的字段，都必须能够被场景系统检索和识别，称为**场景可寻址变量**。

“可寻址”不等于“全部可以无约束直接修改”。变量按照角色分为：

- **直接可控外生变量**：战争、制裁、政策利率、商品冲击、全球指数、加密货币、期货、房地产等，用户可直接设定；
- **条件可控状态变量**：风险偏好、流动性、产能利用率、资本吸纳等，通常由上游推导，但压力测试可人工覆盖；
- **内生推导变量**：企业成本、利润预期、资金净流入等，默认由因果规则计算，只允许带理由的测试性强制覆盖；
- **最终输出变量**：牛熊概率、行业收益、个股路径、日K和逐笔数据，正常因果模式下不可直接设定，仅模块测试模式允许注入。

每次用户干预必须形成正式干预合同，至少包含：变量、作用对象、开始和结束时间、绝对值/增量/涨跌幅/开关/路径/分布、持续方式、是否向下传播、优先级、冲突策略、人工理由和审核状态。

系统支持三类运行模式：

```text
CAUSAL_SCENARIO      只操纵上游现实变量，完整因果传播
DIRECT_MARKET_SHOCK  直接注入指数、加密货币、期货或商品市场扰动
HYBRID               上游事件与直接市场冲击同时存在
```

直接市场冲击模式是专门的测试入口，不代表系统已经解释其上游原因，必须标记为外生冲击，不能伪装成完整因果结论。

用户修改一个变量后，其他变量按照已审核规则联动。大语言模型和专家可以提出候选因果规则，但规则默认只能进入 `DRAFT` 或 `PENDING_REVIEW`，必须经人工核验后才能 `APPROVED/ACTIVE`。

冲突处理优先级为：

```text
数学/物理/会计硬约束
> 交易制度
> 用户明确场景输入
> 已批准因果规则
> 模型估计
> 专家判断
> 代理变量
> 默认值与随机采样
```

发生硬冲突时默认拒绝运行；若为软件异常测试强制运行，必须标记 `CAUSAL_CONSISTENCY_BROKEN`，不得作为有效因果仿真结果。


### 领域抽象与变量命名职责

项目所有者负责提出业务意图、现实场景和使用要求；项目经理负责把自然语言需求抽象成稳定的领域对象、字段、枚举、关系和接口。

用户举例不等于项目必须建立同名字段。正式建模优先采用：

```text
通用实体
+ 通用实体关系
+ 通用事件
+ 事件对实体的影响
+ 数量化暴露
+ 已审核因果规则
```

#### 布尔开关的边界

0/1或布尔值适合回答：

- 事件是否发生；
- 港口是否关闭；
- 工厂是否停产；
- 某项影响是否生效。

但布尔值不能单独表达：

- 影响程度；
- 损失比例；
- 持续时间；
- 恢复周期；
- 替代能力；
- 对不同实体的不同后果。

因此正式模型采用：

```text
开关控制是否发生
事件类型说明发生什么
目标实体说明影响谁
影响类型和度量说明损害哪一方面
连续数值或区间说明影响多大
时间字段说明持续和恢复多久
暴露关系与因果规则说明后续如何传播
```

#### 两个典型例子

“仓库损坏”通常不是独立顶级变量，而是：

```text
事件：空袭/火灾/洪水
目标实体：仓库
影响类型：物理损坏或库存损失
影响度量：库存损失率、停运天数、恢复天数
```

“支柱产业受损”拆分为：

```text
国家—行业的结构关系：该行业是否为支柱产业、GDP/出口/财政占比
+
事件对行业的动态影响：产能、产量、物流、成本或收入变化
```

该原则用于防止字段字典因场景例子无限膨胀，并确保未来扩展不推倒重来。

## 全网应用统计证据发现与因果知识治理

本项目长期建设 `Applied Statistical Evidence Intelligence (ASEI)`。

该能力不是“只读论文”，也不限于心理学或学术期刊。系统应当从全网发现所有具有可核验统计方法和研究价值的公开材料，包括论文、预印本、工作论文、政府统计、监管和国际组织报告、智库和研究机构报告、行业白皮书、调查、民调、杂志数据文章、学者刊物、数据新闻、复现、评论、勘误和撤稿。

长期目标：

```text
公开统计来源
→ 统计主张
→ 标准变量和实体
→ 效应量、方向、不确定性、样本和适用条件
→ 研究设计与因果识别等级
→ 支持、反证、复现和撤稿关系
→ 候选字段、候选特征、候选关联规则或候选因果规则
→ 人工审核与本项目复核
→ 批准、启用、监控或退役
```

硬性判断：

- 所有基于统计的公开结果都有“候选研究价值”，但证据质量和可用权限不同；
- 统计相关不等于因果，预测有效也不等于因果成立；
- 必须区分描述、相关、预测、准因果、随机实验因果和数学/物理/会计硬约束；
- 统计显著不能替代效应量、置信区间、样本外验证、经济意义和可迁移性；
- 反证、复现失败、方法批评、勘误和撤稿与正面证据具有同等保存义务；
- 大语言模型和自动化工具只能提出候选主张、字段和规则，不能自动批准和启用；
- 外部结论进入正式字段、特征、CWMS规则或交易逻辑前，必须经过方法审核、领域审核和本项目数据复核；
- 涉及用户、作者、专家和机构的画像必须基于匿名ID、行为观测和时点快照，不能把群体统计规律永久贴到具体个人；
- 内容、作者、画像、统计主张、特征版本和可信度评估必须一一可追溯；
- 全网发现必须遵守版权、来源许可、网站条款、隐私最小化和访问频率限制。

该能力属于长期横向知识基础设施，不改变当前MVP顺序。当前先完成权威设计和字段治理边界；正式抓取、批量解析、证据图谱和自动监控由后续独立任务小步实现。

专项权威文件：

- `APPLIED_STATISTICAL_EVIDENCE_INTELLIGENCE.md`

## 机构化研究生产线与券商级执行安全

本项目长期借鉴大型量化机构和券商的公开方法，但只学习可迁移的方法和工程纪律，不照搬其硬件规模、组织层级和高频系统复杂度。

### 1. 从量化机构学习什么

大型量化机构最值得学习的不是某个神秘策略，而是把研究变成一条可重复、可比较、可淘汰的生产线：

```text
数据和研究问题
→ 实验登记
→ 因子与模型研究
→ 样本外和滚动验证
→ 交易成本与容量验证
→ CWMS情景压力测试
→ 目标组合与风险门控
→ 模拟盘
→ 有限实盘
→ 正式实盘
→ 持续监控、暂停或退役
```

硬性原则：

- 每次研究必须先登记研究问题和可证伪假设；
- 保存数据版本、字段版本、代码提交、配置哈希和随机种子；
- 因子、模型和策略必须与明确基准比较；
- 同时保存成功与失败实验；
- 禁止只保留漂亮结果；
- 单个因子或模型分数不得直接决定最终仓位；
- Alpha只是输入，组合、风险、成本和容量共同决定是否可用；
- 研究、回测、日报和未来实盘必须尽量使用同一套字段和业务语义；
- 自研基础设施的目标是提高实验吞吐、稳定性和可复现性，而不是炫耀技术规模。

### 2. 研究生命周期

因子、模型、策略和风险规则采用统一晋级状态：

```text
IDEA
→ RESEARCH
→ VALIDATED
→ PAPER
→ LIMITED_LIVE
→ LIVE
→ SUSPENDED / RETIRED
```

晋级不是凭一次高收益回测，而应至少考察：

- 数据质量和可用时间正确；
- 无未来数据泄漏；
- 样本外和Walk-forward稳定；
- 多市场状态表现；
- 交易成本后仍有效；
- 流动性和容量可接受；
- CWMS合成市场和极端场景不过度脆弱；
- 与现有因子、策略和组合的相关性及增量价值；
- 失败时能够明确停用和回滚。

### 3. 从券商学习什么

券商最值得学习的是交易状态清楚、资金持仓可核对、异常可以安全停止。

未来自动交易链必须是：

```text
策略和模型
→ 目标组合
→ 订单生成
→ 独立交易前风控
→ 券商网关
→ 订单状态机
→ 成交回报
→ 实际持仓和资金
→ 账实核对
```

禁止策略直接向券商通道发送订单。

### 4. 订单和执行安全

未来正式订单必须具备：

- `client_order_id`和幂等键，防止网络重试导致重复下单；
- 完整订单状态机；
- 交易前资金、持仓、T+1、涨跌停、停牌、单股和行业上限检查；
- 分级停机能力：暂停开仓、只允许卖出、撤销活动订单、停止策略、停止账户、全部停止；
- 人工干预记录；
- 订单、成交、持仓、现金和费用的每日账实核对；
- 核对失败时可阻止新交易；
- 所有关键动作保留审计记录。

### 5. 环境隔离和版本回滚

至少区分：

```text
RESEARCH
BACKTEST
PAPER
LIVE
```

研究参数和代码不得未经验证直接进入实盘。

重要版本应遵循：

```text
研究与单元测试
→ 历史回测
→ CWMS压力测试
→ 模拟盘
→ 有限资金验证
→ 正式部署
```

并保留：

- 代码版本；
- 配置版本；
- 数据版本；
- 部署版本；
- 可回滚稳定版本；
- 备份和恢复演练记录。

### 6. 个人版实施边界

当前阶段不建设微秒级低延迟、FPGA、大型GPU集群、多机房双活、复杂微服务和高频做市。

当前优先吸收以下机构级纪律：

```text
研究可复现
数据可追溯
失败实验可查询
风险可否决
回测足够真实
订单不会重复
系统能够停机
账实能够核对
版本能够回滚
```

这些长期要求不得阻塞当前DolphinDB数据适配和质量验收，但所有后续模块应预留相应接口和状态。

## 教学式代码注释永久原则

本项目全部人工代码必须使用教学式前置注释。每个逻辑代码块先用一句话概括，再以`# -`分点说明API、参数、数据变化、类型、维度、范围和状态，并单独解释“为什么这样写”。

关键常量必须记录来源、单位、有效范围和实例。docstring可以保留，但不得代替代码块前置`#`注释。

```text
CODE_COMMENTING_STANDARD.md
```

机器政策：

```text
configs/engineering/code_comment_policy_v0.json
```
