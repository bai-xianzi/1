# A股量化投资辅助操作系统——开发指导原则

## 1. 开发总原则

本项目必须从大局观出发，在通用、稳定、可扩展的总体架构内逐步实现功能。

```text
稳定骨架
→ 最小功能跑通
→ 真实数据验证
→ 小步扩展
→ 多次二次开发
→ 尽量不推倒重来
```

## 2. 数据系统总架构

```text
交易所、厂商、公开API、数据库、文件、自定义数据
→ Raw原始层
→ 数据源适配器与领域语义适配器
→ 统一字段字典和标准对象
→ 数据质量、血缘、可见时间和来源冲突处理
→ 标准数据服务
→ 市场状态、风险、因子、模型、组合、回测和报告
```

### 硬性要求

- 不能围绕某一家厂商或某一张表设计整个系统；
- 当前不用的数据源可以不实现，但接口和模块归属要留有余地；
- 原始字段必须保留；
- 标准字段是下游模块唯一允许依赖的金融语义；
- 新来源优先通过新增适配器和映射文件接入；
- 不能因为开发方便改变金融定义；
- 不确定的单位、复权和时间含义必须标记待确认。

## 3. 标杆借鉴方法

### 开源项目

研究源码、接口、算法、测试、Issue、文档和许可证，然后形成语言无关的功能规格，再使用本项目技术栈实现。

### 闭源项目

研究用户手册、API、字段词典、参数、返回结构、演示、培训资料、白皮书和更新日志。

必须区分：

- 公开确认事实；
- 有证据支持的合理推断；
- 无法确认的内部实现；
- 本项目自主设计决定。

## 4. 渐进式演进

应尽量稳定：

- 统一字段字典；
- 标准领域对象；
- 模块输入输出；
- 数据时间和复权原则；
- 市场状态—风险—仓位—策略—组合主链。

可以逐步替换：

- 算法；
- 模型；
- 阈值；
- 报告；
- 前端；
- 券商执行方式。

## 5. 当前开发起点

不是直接围绕现有数据库写业务逻辑，而是：

```text
定义通用数据契约
→ 定义分层、映射、质量和血缘
→ 实现DolphinDB日K首个适配器
→ 用真实数据验收通用链路
→ 扩展基本面和七类快照
```

## 6. 每次新增模块前必须回答

1. 属于总体结构哪一层？
2. 当前最小可运行版本是什么？
3. 将来可能扩展什么？
4. 现在要预留什么边界？
5. 是否依赖标准字段而不是来源字段？
6. 是否保留原始数据和血缘？
7. 是否破坏已有功能？
8. 能否新增或替换而不是整体重写？
9. 如何测试旧功能继续可用？


## 7. 因果行情仿真开发约束

开发行情生成、市场预测、因子挖掘和策略回测时，必须共用CWMS的标准变量、因果规则和合成行情。

禁止：

- 独立随机生成牛熊标签；
- 为单个策略临时造一套不受约束的行情；
- 先画K线再编因果解释；
- 将场景假设或专家判断当作真实观测；
- 让合成行情使用另一套字段名称；
- 让逐笔、分钟和日K互相矛盾。

必须：

- 先形成场景和上游变量；
- 传播因果和企业暴露；
- 形成参与者行为、资本迁移和市场状态；
- 最后生成具体路径；
- 记录因果血缘、场景、规则、版本和随机种子；
- 使用同一合成数据集验证预测、因子、多因子、策略、组合和风险。

## 8. 用户场景变量开发约束

所有影响行情的标准字段最终都要能够被场景控制系统检索，但必须区分直接控制权限：

- 外生世界变量：允许直接设置；
- 条件状态变量：允许场景覆盖；
- 内生变量：默认由因果规则推导；
- 行情输出：仅模块测试允许强制注入。

开发任何可控变量时必须定义：字段引用、作用对象、控制方式、合法范围、时间路径、传播模式、冲突策略、默认优先级和人工说明。

专家或大语言模型提出的新逻辑必须以候选规则保存，人工审核前禁止启用。直接指数、加密货币、期货或商品冲击必须标注为直接市场扰动，不能伪装成已解释的完整因果链。


## 8. 自然语言需求到领域模型的转换

项目所有者提出场景和目标，项目经理负责确定技术模型和字段命名。

新增变量前必须先判断需求属于：

- 通用实体；
- 实体关系；
- 通用事件；
- 事件影响；
- 暴露关系；
- 因果规则；
- 真正稳定的连续状态变量。

不得将每个业务例子机械做成字段。

布尔值用于门控，枚举用于类型，连续数值用于程度，时间字段用于持续和恢复，关系对象用于说明影响路径。


## 9. 需求判断与项目经理职责

甲方提出的是业务意图和使用要求，项目经理负责决定技术表达。

每次收到新需求时，必须先回答：

1. 用户真正想解决的问题是什么？
2. 这是新字段、新对象、新关系、新事件、新规则，还是已有模型的参数？
3. 能否使用现有通用领域模型表达？
4. 直接照用户举例实现会不会造成字段爆炸或未来重构？
5. 最小可运行实现是什么？
6. 哪些边界现在必须稳定，哪些实现可以后续替换？
7. 需要向甲方说明哪些取舍和风险？

项目经理有责任纠正不合理的技术假设，而不是把用户的每句话直接变成代码。


## 10. 机构化研究和生产安全约束

### 10.1 实验必须登记

任何正式因子、模型、策略、组合或风险规则实验，必须登记：

- 研究问题与假设；
- 数据和字段字典版本；
- 代码提交和配置哈希；
- 训练、验证、测试区间；
- 基准；
- 成本和容量模型；
- 主要指标；
- 失败原因和结论；
- 下一步晋级或退役动作。

负面结果不得删除。

### 10.2 晋级必须经过门槛

不得直接从研究回测进入实盘。标准顺序：

```text
RESEARCH → VALIDATED → PAPER → LIMITED_LIVE → LIVE
```

每一步都必须有明确验收标准。

### 10.3 执行必须安全隔离

未来接券商接口时：

- 目标组合与订单分离；
- 订单前必须经过独立风控；
- 必须具有幂等键；
- 必须实现订单状态机；
- 必须实现分级停机；
- 必须做订单、成交、持仓、现金和费用核对；
- 研究、回测、模拟盘和实盘配置必须隔离；
- 发布必须保留回滚方案。

当前人工交易阶段只预留标准对象和接口，不提前开发完整自动交易系统。

## 11. 全网应用统计证据与因果知识开发约束

本项目长期具备从全网发现应用统计成果并转化为候选知识资产的能力。来源不限于论文，也包括预印本、工作论文、政府统计、智库报告、行业研究、调查、民调、杂志数据文章、学者刊物、复现、评论、勘误和撤稿。

正式链路：

```text
来源发现
→ 来源身份、版本和许可确认
→ 统计主张提取
→ 变量与标准字段映射
→ 研究设计和因果识别等级判断
→ 效应量、不确定性、样本和适用范围提取
→ 支持、反证、复现和撤稿关系
→ 候选字段、特征、关联规则或因果规则
→ 人工审核
→ 本项目数据复核和样本外验证
→ 批准、启用、监控或退役
```

硬性要求：

- “基于统计”只表示具有候选研究价值，不表示结论已经可靠；
- 必须区分描述性统计、相关关系、预测关系、准因果、随机实验因果和硬约束；
- 统计显著不等于效应大、经济意义强、可迁移或可交易；
- 每条主张必须保留来源、版本、发布时间、获取时间、样本、方法、效应量、不确定性、适用范围和限制；
- 必须主动保存反证、复现失败、方法批评、勘误和撤稿；
- 外部结论不能直接改写字段字典或生产规则，只能生成候选项；
- 自动化和大语言模型可以提取、归一化和生成审核任务，但不能自动批准字段、因果规则和交易逻辑；
- 涉及人的统计规律只能用于概率性、时点化和可撤销的行为画像，不得形成永久人格标签；
- 评论、帖子、作者画像和可信度评估必须通过稳定匿名ID和时点快照一一关联；
- 全网发现必须遵守版权、网站条款、访问频率、隐私最小化和来源许可；
- 当前不建设大规模爬虫，先稳定合同、证据对象和审核边界。

专项权威文件：

- `APPLIED_STATISTICAL_EVIDENCE_INTELLIGENCE.md`

## 新Provider开发强制流程

```text
登记目标
→ 实际SDK与授权发现
→ 许可证和认证边界
→ 能力合同
→ Raw适配
→ Canonical映射
→ 时点、质量和血缘
→ 合同测试
→ 真实验收
→ 人工激活
```

禁止凭供应商名称或宣传材料假设实际能力。禁止在核心模块直接导入厂商SDK。禁止把账号、密码、Token或券商账户信息写入Git。

当前单机开发默认Provider、CPU和数据库并行度均不超过2；大于5GB下载需要人工覆盖；不自动导入35GB分钟数据。

## 开发前置复用调查

每个新功能开工前必须记录：

- 项目内搜索结果；
- 官方SDK或标准；
- 成熟开源候选；
- 许可证和安全；
- 维护和兼容；
- 语义缺口；
- 复用、薄适配或自研决定。

没有完成调查时，不得创建重复Adapter、客户端、协议实现、重试器、分页器、缓存器或配置系统。

## 教学式注释开发流程

每个开发任务固定执行：

```text
复用调查
→ 代码与教学式注释同步设计
→ 注释审计
→ 专项测试
→ 完整测试
→ 编码审计
→ Git空白检查
→ 用户功能补充确认
→ 用户提交确认
→ Git提交和推送
```

代码能够运行不等于任务完成。代码、教学式注释、测试和权威文档同时合格后才算完成。

修改旧代码时，必须同步补齐该逻辑区域的注释，不得扩大历史注释债务。
