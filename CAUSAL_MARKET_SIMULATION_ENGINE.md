# 因果驱动的世界—经济—行为—资本—市场仿真引擎

英文暂定：`Causal World-to-Market Simulation Engine (CWMS)`

## 1. 定位

本引擎不是随机画K线工具。它首先使用标准字段字典中的现实、人工、估计、代理、专家和场景变量，按因果规则推导世界状态、企业经营、参与者行为、资本迁移和市场牛熊结构，然后才随机生成具体日K、分时、逐笔和订单簿路径。

```text
世界与场景变量
→ 因果硬逻辑和条件逻辑
→ 企业与行业暴露
→ 企业收入、成本、产能、库存和现金流
→ 居民、机构、外资、游资等参与者行为
→ 股票、房地产、存款、理财、债券、黄金等资本迁移
→ 牛市、熊市、震荡、危机、流动性和行业结构
→ 受约束的具体行情路径
```

## 2. 不可违反的原则

- 不是先生成行情再编故事。
- 大体牛熊、流动性和行业结构不是随机生成。
- 结构字段不能随机。
- 内生变量不能脱离父变量独立随机。
- 事件不能直接跳到股票涨跌，必须经过企业和资本传导。
- 硬恒等、硬约束和交易制度不可违反。
- 条件因果必须保存适用和失效条件。
- 合成、人工、专家和推导值不得冒充真实数据。
- 每条路径必须可解释、可追溯、可复现。

## 3. 随机性的正确位置

```text
因果结构：原则上不随机
参数强度与时滞：在合理区间内采样
市场微观路径：在牛熊和行业约束下随机
```

同一熊市场景可以生成缓慢阴跌、急跌反弹、结构性分化等多条路径，但不能无原因生成持续全面大牛市。

## 4. 通用验证对象

本引擎是横向研究验证底座，服务于：

- 行情预测和牛熊识别；
- 单因子挖掘；
- 多因子选股；
- 个股评分和排名；
- 策略回测；
- 组合构建；
- 风险仓位和策略门控；
- 交易执行边界；
- 报表和整套软件回归测试。

## 5. 标准数据输出

合成行情与真实行情使用同一标准字段。下游模块不应因数据来自真实市场或合成市场而改写业务逻辑。

额外保留：

- `simulation_run_id`
- `scenario_id`
- `monte_carlo_path_id`
- `causal_trace_id`
- `generator_version`
- `rule_version`
- `random_seed`
- `configuration_hash`

## 6. 分层设计

1. 标准变量与输入方式；
2. 因果规则和专家硬逻辑；
3. 企业行业暴露关系；
4. 世界和经济状态传播；
5. 参与者行为与资本迁移；
6. 市场状态和行业个股冲击；
7. A股市场制度规则；
8. 日K、分时、逐笔和订单簿路径生成；
9. 一致性、约束和因果血缘校验；
10. 对因子、模型、策略、组合和风险系统的统一评估。

## 7. 渐进式实施

### V1
- 少量宏观、政策、资金变量；
- 牛、熊、震荡和危机状态；
- 行业级传导；
- A股日K生成；
- 可复现种子和规则校验；
- 验证市场状态模型、因子和基础策略。

### V2
- 战争、能源、物流和企业地域暴露；
- 专家逻辑注册与历史评分；
- 多资产资本迁移；
- 多股票和行业相关性。

### V3
- 居民与机构行为；
- 房地产、存款、理财、债券、黄金和海外资产；
- 公司供应链、客户、仓库和工厂冲击；
- 分钟与集合竞价。

### V4
- 逐笔成交；
- 订单簿、撤单和排队；
- 微观流动性和执行压力测试。

## 8. 与真实回测的关系

```text
真实历史回测
+ 样本外和滚动验证
+ 因果合成市场Monte Carlo
+ 极端情景压力测试
= 完整验证体系
```

合成行情不能单独证明现实盈利能力，但可以发现模型、因子、策略和风险系统的脆弱性。

## 9. 用户场景控制平面

用户不是只能选择几个固定模板，而应能够在标准字段字典中搜索所有市场影响变量，查看定义、单位、合法范围、上游、下游、规则、置信度和直接控制权限，并决定是否施加干预。

### 场景可寻址与直接可修改的区别

- 所有市场影响变量都应当**场景可寻址**；
- 只有外生变量默认允许直接修改；
- 内生变量通常由规则推导；
- 输出变量正常模式不可直接修改；
- 模块测试可强制覆盖，但必须记录原因和破坏的因果链。

### 干预合同

每次控制至少记录：

- `intervention_id`
- `scenario_id`
- 标准字段引用
- 目标国家、地区、行业、公司、证券或市场
- 设定绝对值、增量、涨跌幅、开关、路径或概率分布
- 开始、结束、持续和爬坡方式
- 是否自动传播
- 冲突策略和优先级
- 是否强制覆盖
- 人工理由、创建者和审核状态

## 10. 三种运行模式

### 因果场景模式

只设置战争、政策、居民收入、商品供给、产业受损等上游变量，系统完整传播到市场。

### 直接市场扰动模式

允许用户直接设置全球指数、A股大盘、加密货币、期货、商品或外汇涨跌，用于测试跨市场传染、因子、策略和风险系统。该冲击必须标记为 `DIRECT_MARKET_SHOCK`，不宣称已经解释上游原因。

### 混合模式

同时设置现实事件和市场冲击，从当前已知状态继续向行业、企业和A股传播。

## 11. 因果规则提出、审核与启用

专家、历史统计和大语言模型都可以提出候选逻辑，但不能直接成为生产规则。

```text
提出候选规则
→ 结构化原因、结果、条件、时滞和强度
→ 查找公开证据、历史案例和反例
→ 人工判断硬恒等/硬约束/硬方向/条件/概率/相关
→ 历史校准与小样本回放
→ 人工批准
→ ACTIVE
```

大语言模型生成的规则必须标记来源，未经人工批准只能处于 `DRAFT` 或 `PENDING_REVIEW`。

## 12. 冲突检测与优先级

系统必须识别用户输入与硬逻辑、交易制度、其他干预和推导值之间的冲突。

默认优先级：数学、物理和会计约束 > 交易制度 > 用户上游输入 > 已批准因果规则 > 模型和专家估计 > 代理与默认采样。

硬冲突默认拒绝运行。强制运行只用于软件异常测试，并必须标记 `CAUSAL_CONSISTENCY_BROKEN`，不能进入有效因果研究样本。

## 13. 字段引用与重复名称

统一字段在不同领域对象中可能重复使用相同名称，因此机器引用优先采用：

```text
domain_code.canonical_name
```

例如 `market_daily.close_raw_cny`。只有在全局无歧义或调用上下文已经声明领域时，才允许使用裸字段名。

## 14. 当前覆盖边界

当前字典已具备CWMS框架，但 `simulation_variables.yaml` 仍是代表性样例，不表示945个字段已经全部完成仿真合同。后续必须按领域逐步完成覆盖审计，不能宣称“字段已定义”等于“因果和生成规则已完成”。


## 15. 通用事件—实体—影响模型

CWMS不为每个现实场景创建专用变量。场景首先映射为以下稳定对象：

```text
WorldEntity
→ EntityRelation
→ WorldEvent
→ EventImpact
→ EntityExposure
→ CausalRule
```

### WorldEntity

表示国家、行业、公司、工厂、仓库、港口、航线、能源设施和市场等对象。

### EntityRelation

表示稳定关系，例如：

- 仓库属于哪个公司；
- 公司依赖哪个港口；
- 某行业是不是某国支柱产业；
- 某行业占GDP、出口或财政收入多少。

### WorldEvent

表示发生了什么。`event_active_flag`只负责门控，不描述损害程度。

### EventImpact

表示一个事件对一个实体的具体后果。一个事件可产生多条影响：

```text
空袭
→ 某仓库库存损失40%
→ 停运30天
→ 预计90天恢复
```

### EntityExposure

表示主体对地区、供应链、商品、基础设施等的敏感度和脆弱性，不等同于已发生的损害结果。

### 建模规则

- 0/1只表达是否发生或是否生效；
- 类型使用枚举；
- 程度使用连续数值或区间；
- 时间使用开始、结束、停运和恢复时长；
- 支柱产业属于结构关系；
- 支柱产业受损属于动态事件影响；
- 不为“火灾烧仓库”“洪水淹仓库”“战争炸仓库”分别建立字段。


## 16. 场景需求的建模决策

用户可以用自然语言描述“仓库被炸”“支柱产业受损”等场景，但系统实现不能机械创建同名变量。

项目经理必须先判断：

- 是否只是创建一个通用事件；
- 是否需要指定一个已有或新建实体；
- 是否需要增加实体关系；
- 是否只是新增一条事件影响；
- 是否需要补充暴露权重；
- 是否需要新增或审核因果规则；
- 只有通用模型无法表达时，才新增稳定标准变量。

前端可以向用户提供开关、滑块和场景模板，但后台必须保存完整的事件、实体、影响、时间和传播信息。
