# 复用优先开发政策（最高优先级工程门禁）

> 政策级别：P0。
> 适用范围：所有新功能、新模块、新Provider、新基础设施、重构和重大技术选型。
> 核心结论：**先借鉴、再复用、优先薄适配，确实没有可用方案时才允许最小范围自研。**

## 1. 政策目的

本项目不以“所有代码都由自己重新编写”为目标，也不把重复造轮子视为自主可控。

开源社区、官方SDK、行业标准和成熟工程已经解决并验证过的通用问题，应优先学习和复用。项目有限的开发资源应集中在真正具有项目独特价值的部分，例如A股语义治理、用途级Readiness、市场状态—风险—仓位主链、真实A股成交约束、应用统计证据治理和CWMS因果仿真。

## 2. 最高优先级决策顺序

```text
项目仓库已有能力
→ 供应商官方SDK、正式API和官方示例
→ 已公开且稳定的行业标准
→ 许可证明确、持续维护、测试充分的成熟开源组件
→ 对成熟能力建立薄Adapter、Bridge、Facade或Provider插件
→ 借鉴成熟架构、接口、测试和失败经验后局部补充
→ 前述方案确实无法满足时，才允许最小范围自研
```

完全独立开发是最后选项，不是默认选项。

## 3. 开工前强制复用门禁

任何功能进入详细设计或编码前，必须完成并记录以下检查：

1. 项目仓库中是否已经存在同类能力；
2. 是否存在官方SDK、正式API、标准协议或官方示例；
3. 是否存在成熟、活跃、许可证兼容的开源实现；
4. 哪些部分可以直接复用，哪些部分只需薄适配；
5. 第三方方案是否满足Windows、当前Python版本和单机资源边界；
6. 是否满足A股语义、Point-in-Time、血缘、Canonical和Readiness要求；
7. 是否会把核心系统绑定到单一供应商或私有类型；
8. 如果决定自研，现有方案具体缺口是什么，自研最小范围是什么；
9. 第三方组件未来如何替换、升级、回滚和隔离；
10. 测试、许可证、安全和维护责任如何落实。

未完成复用调查和决策记录时，不得进入大规模编码。

## 4. 功能需求与实现方式的关系

用户提出的是要实现的业务能力，不代表必须自己从零编写底层实现。

正确处理方式是：

```text
识别真实业务能力
→ 查找社区和官方成熟成果
→ 分离通用能力与项目独特能力
→ 通用能力优先复用
→ 项目独特能力保持自主语义和稳定接口
```

复用第三方能力不改变本项目对Canonical语义、数据门禁、风险控制、可审计性和供应商中立的要求。

## 5. Adapter、Provider和基础设施专门规则

对以下通用问题必须优先复用，不得轻易重新实现：

- 数据库客户端、连接池和会话管理；
- 数据厂商和券商官方SDK；
- 认证、重试、限流、分页和订阅；
- 文件格式、序列化、压缩和校验；
- 日志、配置、缓存和任务调度；
- 测试框架、统计模型和常用算法；
- 图表、报表和基础Web组件；
- 标准回测或机器学习基础设施中已经成熟的通用部分。

例如：

```text
已有DolphinDBDataSourceAdapter
→ 建立ProviderPlugin薄桥接
→ 复用查询、会话和错误处理
→ 不重新实现DolphinDB客户端
```

Wind、iFinD、银河证券星耀数智、QMT、PTrade等平台接入时，应优先使用正式SDK或API，并把厂商差异限制在插件实现层。

## 6. 第三方组件评估

引入前至少检查：

- 官方来源和代码仓库；
- 当前版本、维护活跃度和兼容版本；
- 许可证以及商业使用、缓存和再分发边界；
- 已知安全问题；
- Windows、Python和当前硬件兼容性；
- 内存、磁盘、网络和并发要求；
- 数据语义、时点、单位和精度；
- 是否可测试、可替换、可回滚；
- 社区质量、Issue处理和发布稳定性。

复用优先不等于盲目引入。质量不可靠、许可证不清、维护停止或边界不兼容的方案不得直接进入生产链。

## 7. 禁止事项

- 未搜索项目现有代码就创建重复模块；
- 未调查社区和官方方案就直接从零开发；
- 为了“全部自己写”重写成熟协议、客户端和通用算法；
- 复制未知许可证、来源不明或无法审计的代码；
- 根据博客、宣传材料或猜测伪造SDK能力；
- 把第三方私有类型和字段传播到上层研究代码；
- 没有替换边界地深度绑定单一供应商；
- 仅因为开源就跳过安全、许可证、语义和数据质量审查；
- 为引入大型框架而破坏当前个人电脑资源边界；
- 把“参考了某项目”写成无法追溯来源的口头结论。

## 8. 自研批准条件

自研必须同时记录：

```text
明确业务需求
已调查的项目内、官方和开源候选
候选方案比较
无法直接复用或薄适配的具体原因
许可证、安全、语义或资源阻断
自研的最小边界
维护负责人和长期成本
测试与真实验收计划
未来替换、兼容或迁移路径
```

缺少任何一项时，自研决定不得通过。

允许自研的典型情况包括：

- A股制度、金融语义或时间可见性要求无法由现有方案满足；
- 现有组件无法接入Canonical和Readiness边界；
- 许可证、安全、资源或维护状态不可接受；
- 功能属于本项目真正独特的核心能力；
- 通过薄适配仍会造成严重语义泄漏或供应商锁定。

## 9. 每个任务必须留下的复用决策记录

任务说明或验收报告至少包含：

```text
需求名称
项目内搜索结果
官方方案候选
成熟开源候选
许可证与维护结论
直接复用部分
薄适配部分
必须自研部分
最终选择及理由
替换和回滚方案
```

小任务可以使用简表，大型功能应形成独立调研或ADR。没有该记录，任务不能进入“实现完成”。

## 10. 与全供应商适配的关系

全适配不意味着为每个供应商从零开发一套系统。

正确方式是：

```text
通用协议
+
官方SDK或成熟客户端
+
薄Provider插件
+
Canonical映射
+
质量、血缘和时点门禁
```

## 11. 阶段完成标准

涉及新能力的阶段完成前，必须确认：

- 已完成复用调查；
- 没有重复实现仓库已有能力；
- 第三方来源、版本和许可证可追溯；
- 私有类型没有越过插件或Canonical边界；
- 自研部分已缩小到必要范围；
- 专项测试和回归测试通过；
- 权威文件、任务记录和验收报告同步；
- 经过验证的Git提交已推送到GitHub `main`。
