# SYSTEM_ARCHITECTURE.md

# A股量化投资辅助操作系统总体结构图（V9项目权威版）

> 本文件是 `wjx--` 项目的总体结构权威文件。  
> 后续凡是项目长期记忆、模块边界、开发顺序或重大设计发生变化，都必须同步检查并更新本文件。  
> 总原则：**逻辑可以机构级，实现必须个人级。**

---

## 1. 项目当前目标

先稳定形成以下闭环：

```text
数据
→ 市场状态
→ 风险仓位
→ 股票排名
→ 真实回测
→ T日计划
→ T+1竞价确认
→ 人工交易
→ 结果反馈
```

## 3. 横向贯穿能力



### 需求治理、领域抽象与技术决策

```text
甲方提出目标、场景和约束
→ 项目经理识别真实业务问题
→ 检查逻辑冲突和过度设计
→ 抽象领域对象、字段、关系和接口
→ 说明取舍与未来影响
→ 甲方确认
→ Codex按任务包实现
→ 项目经理验收
```

硬性要求：

- 甲方负责需求、优先级和验收，不负责决定具体字段和代码结构；
- 项目经理不得机械接受自然语言例子；
- 新场景优先判断是否可由通用实体、关系、事件、影响、暴露和因果规则表达；
- 重大设计必须说明为什么这样设计、替代方案是什么、未来扩展成本如何；
- Codex和Trae只执行，不拥有架构与金融语义决策权。




### 全网应用统计证据、因果知识与特征治理

```text
论文 / 预印本 / 工作论文 / 政府统计 / 智库报告
/ 调查民调 / 行业白皮书 / 杂志数据文章 / 数据新闻
/ 复现 / 反证 / 勘误 / 撤稿 / 数据与代码
→ 来源身份、版本、许可和available_at
→ 统计主张、变量、样本、方法和效应量提取
→ 标准字段、标准实体和关系映射
→ 描述 / 相关 / 预测 / 准因果 / 因果等级
→ 支持、反证、复现和撤稿证据图
→ 候选字段、特征、关联规则或因果规则
→ 方法审核、领域审核和本项目复核
→ APPROVED / ACTIVE / DISPUTED / REJECTED / RETIRED
```

硬性要求：

- 任何有统计方法的公开材料都可以进入候选库，但不得因“有数字”而自动获得高可信等级；
- 来源形式不决定结论，研究设计、样本、测量、效应量、不确定性、可复现性和外部有效性决定证据等级；
- 相关性不得自动升级为因果，预测关系不得自动描述为机制；
- 支持证据、反证、复现失败、勘误和撤稿必须共同保存；
- 新变量优先映射已有标准字段，不能为每篇文章或每条结论创建专用字段；
- 自动化和大语言模型只生成待审核记录，不能自动启用正式字段、因果规则、因子或交易逻辑；
- 涉及人的行为画像必须匿名、时点化、概率化和可撤销，具体内容必须与作者ID、画像快照和证据版本一一关联；
- 全网发现遵守版权、许可、网站条款、隐私最小化和访问频率限制；
- 当前只建立权威合同和模块边界，不提前建设大规模爬虫。

专项权威文件：

- `APPLIED_STATISTICAL_EVIDENCE_INTELLIGENCE.md`


### 机构化研究生产线与实验生命周期

```text
研究问题和假设
→ 实验登记
→ 数据/字段/代码/配置版本冻结
→ 因子模型和策略研究
→ 样本外、Walk-forward、成本和容量验证
→ CWMS多场景压力测试
→ 模拟盘
→ 有限实盘
→ 正式实盘或暂停退役
```

硬性要求：

- 成功和失败实验都必须登记；
- 结果必须可以由数据版本、代码提交、配置哈希和随机种子复现；
- 模型和策略不能凭单次高收益回测晋级；
- 研究晋级状态统一为 `IDEA → RESEARCH → VALIDATED → PAPER → LIMITED_LIVE → LIVE`；
- 任一阶段出现数据泄漏、成本后失效、容量不足或风险失控，均可暂停或退役；
- 自研基础设施优先服务实验吞吐、缓存、增量计算和复现，不追求不必要的分布式复杂度。

### 券商级执行安全、账实核对与故障恢复

```text
目标组合
→ 订单生成
→ 独立交易前风控
→ 券商网关
→ 订单状态机
→ 成交/持仓/资金
→ 账实核对
→ 异常阻断、停机和恢复
```

硬性要求：

- 策略不得直接发单；
- 每个订单必须具有幂等键和完整状态；
- 网络超时不得盲目重发；
- 必须支持暂停新开仓、只卖出、撤销活动订单、停策略、停账户和全部停止；
- 订单、成交、持仓、现金和费用必须核对；
- 核对失败时默认阻止新的自动交易；
- 研究、回测、模拟盘和实盘环境必须隔离；
- 重要发布必须经过测试、可回滚并保留审计记录。


### 多源数据适配、统一语义与血缘

```text
交易所 / 数据厂商 / 数据库 / 文件 / 自定义数据
→ Raw原始层
→ 来源与领域适配器
→ 统一字段字典和标准对象
→ 质量、血缘、可见时间和冲突校验
→ 标准数据服务
→ 下游业务模块
```

硬性要求：

- 数据架构不能围绕单一厂商或当前表结构写死；
- 原始数据和来源字段必须保留；
- 每个来源通过独立映射文件进入标准字段体系；
- 同一标准字段允许主来源、备用来源和对账来源；
- 所有转换必须可追溯到来源、映射版本和转换规则；
- DolphinDB是第一个真实适配案例，不是总体架构边界；
- 新增数据源原则上只新增适配器和映射，不修改下游业务模块。

### Codex任务上下文治理

Codex不继承长期聊天上下文。正式开发必须遵循：

```text
长期思想
→ 项目权威文件

→ 测试和验收
```

Codex不得自行决定金融语义、总体架构、开发顺序和正式数据库重建。



### 用户可控变量、干预合同与规则审核

```text
用户选择场景可寻址变量
→ 设置作用对象、时间、幅度和变化路径
→ 权限与合法范围检查
→ 因果冲突检查
→ 已批准规则传播
→ 市场状态与具体行情生成
```

硬性要求：

- 所有影响市场的标准字段最终都要接受“是否可寻址、是否可直接控制、怎样传播”的覆盖审计；
- 直接市场冲击属于外生测试输入，不能冒充完整的上游因果解释；
- 专家和大语言模型只能提出候选规则，未经人工批准不得进入ACTIVE规则集；
- 强制覆盖内生或输出变量必须记录理由、作用范围和被破坏的因果关系；
- 机器字段引用优先使用 `domain_code.canonical_name`，避免跨领域同名歧义。


### 通用事件、实体、关系与影响

现实场景的标准表达为：

```text
通用事件
→ 作用目标实体
→ 事件影响类型与度量
→ 实体结构关系与暴露关系
→ 已审核因果规则传播
```

禁止看到一个场景例子就增加一个同名字段。布尔门控、枚举类型、连续程度、时间周期和关系权重必须分开建模。


### 因果行情仿真与通用研究验证底座

```text

→ 仿真变量合同
→ 因果规则和专家硬逻辑
→ 企业行业暴露
→ 参与者行为与资本迁移
→ 市场大状态
→ 受约束的日K、分时、逐笔和盘口路径
→ 统一验证行情预测、因子、多因子、策略、组合和风险系统
```

这是一项横向能力，不归属于单一策略或单一回测模块。

硬性要求：

- 字段字典中的市场相关变量不得被动等待真实数据源；
- 缺失变量可以人工、场景、代理、专家、估计、推导、隐状态或合成方式激活；
- 变量角色必须区分结构、外生、内生、中间状态和行情输出；
- 大体牛熊和行业结构由因果传播决定；
- 微观行情随机生成必须服从A股制度、因果状态和多周期一致性；
- 每次仿真必须可复现、可解释、可追溯；
- 同一套合成行情可同时验证市场预测、因子、选股、策略、组合和风险模块。


### 数据质量门禁
数据未更新、关键字段缺失、复权失败、覆盖率异常时，停止正式交易清单。

### 时间和未来数据防护


### 运行结果保存
正式运行至少保存：

```text
日期、配置、数据截止日期、市场状态、风险仓位、预测、
候选排名、目标组合、日报、竞价结果和日志
```

### 模块化单体
采用一个项目、多个清晰模块，不采用微服务。

## 复用优先架构约束

新架构层必须优先组合现有模块。旧接口升级为新协议时，默认使用Bridge、Adapter或Facade，不复制旧实现。

```text

```

第三方私有类型和字段不得越过插件与Canonical边界。
