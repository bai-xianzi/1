# 代码教学式注释强制标准

## 权威等级

本文件是项目人工代码注释的最高权威标准。

```text
规则级别：MANDATORY
适用范围：全部人工编写代码
默认状态：不符合即阻断Git提交
替代方式：不允许用docstring替代
```

## 核心规则

每一个人工编写的逻辑代码块之前，都必须使用合法的单行注释符号提供教学式前置说明。

Python和PowerShell统一使用以下结构：

```text
# 本段代码核心功能：先用一句完整的话说明接下来代码解决什么问题。
# - 输入：解释输入值来自哪里、类型是什么、允许范围是什么。
# - 处理：解释API、函数和参数如何改变数据、状态、形状或数值范围。
# - 输出：解释结果类型、维度、单位、状态和后续去向。
# - 常量依据：解释固定值的来源、单位、适用范围和实际示例。
# - 为什么这样写：解释算法原理、运行收益、风险控制或工程取舍。
对应代码
```

## 用户原始六条规则

1. 代码块前必须有一段以单行注释符号开头的多行块说明，第一行概括本段代码核心功能。
2. 使用`# -`无序列表，逐行解释下方API、函数、参数的底层逻辑、数据变化、维度变化和数值范围。
3. 关键常量和固定参数必须补充来源、数据集实例或行业通用值。
4. 必须额外解释“为什么要这么写”，说明算法作用、训练或运行收益和数学原理。
5. 注释必须通俗、可实操；禁止使用三引号docstring代替代码块前置说明。
6. 注释和代码必须一一对应，先注释后代码，排版整齐、分点清晰。

## 一一对应的准确含义

“一一对应”按逻辑代码行执行，而不是对纯格式行机械重复。

必须解释：

- 外部API调用；
- 函数调用及其关键参数；
- 类、函数和方法定义；
- 条件分支、循环和异常处理；
- 数据读取、过滤、聚合、映射和写入；
- 类型、形状、单位、范围或状态变化；
- 安全门禁、许可证门禁和交易门禁；
- 固定阈值、端口、批次、并发、超时和重试参数。

不要求为以下纯格式行单独重复注释：

- 单独的右括号；
- 仅用于换行的逗号；
- 空行；
- Python shebang和编码声明；
- PowerShell `#requires`解释器指令。

## 常量说明

关键常量至少说明四件事：

```text
来源
单位
有效范围
实例或行业通用值
```

例如，`8848`必须说明它是当前本机DolphinDB服务端口，而不是任意数字；`100000`必须说明它来自当前16GB单机资源档案中的无人工覆盖批次上限。

不能把项目特有值伪装成行业标准，也不能为未知厂商SDK能力编造来源。

## 为什么要这样写

“为什么这样写”不能只写“为了方便”或“为了性能”。

合格说明应回答至少一个问题：

- 它避免了什么错误？
- 它保护了什么数据或交易安全边界？
- 它为什么能降低内存、磁盘或网络压力？
- 它为什么能提高数值稳定性、可重复性或可解释性？
- 它为什么便于未来替换Provider或SDK？
- 它对应什么数学或统计原理？

## docstring规则

docstring可以保留，用于IDE、类型系统和API文档。

但docstring不能代替代码前的教学式单行注释。以下形式不合格：

```python
def run():
    """运行任务。"""
```

必须在定义前增加教学式说明。

## 文件类型边界

直接应用：

```text
.py
.ps1
.psm1
.psd1
.sh
.sql
.dos
.bat
.cmd
```

JSON不支持注释，因此不能插入`#`。JSON字段的解释必须写入对应Markdown标准、Schema或机器可读政策说明中。

以下文件不进行批量修改：

- 第三方源码；
- 厂商SDK；
- 自动生成代码；
- 锁文件；
- 虚拟环境；
- 缓存；
- 数据文件；
- 日志；
- 历史归档补丁。

## 修改旧代码的规则

修改任何旧代码时，必须同步补齐该逻辑区域的教学式注释。

重构、格式化、移动或拆分代码时，不得删除仍然有效的教学式注释。

发现旧代码缺少注释时，不得继续扩大未注释代码债务。

## 复用优先关系

复用第三方库或官方SDK不代表可以省略注释。

必须解释：

- 为什么选择该库；
- 关键API做了什么；
- 输入输出如何变化；
- 许可证和安全边界；
- 如何通过薄Adapter隔离第三方私有类型；
- 未来如何替换。

## Git门禁

进入Git提交前必须完成：

```text
教学式注释审计
专项测试
完整测试
编码审计
git diff --check
用户功能补充确认
用户提交确认
```

任何一项未完成时，不得执行`git commit`或`git push`。

## 沙盒验证前置门禁

所有新增或修改的人工代码必须在第一次沙盒语法检查、导入检查或单元测试之前完成教学式前置注释。禁止先生成无注释代码、测试通过后再补注释，也禁止把注释整改推迟到交付包或用户本地。

```text
代码与注释同步设计
→ 注释结构自检
→ 第一次沙盒语法检查
→ 专项与回归测试
→ 全仓库注释审计
→ 交付
```

沙盒中被测试的代码和最终交付代码必须是同一份已经完成注释的代码。注释审计失败时，代码即使能够运行也不得进入交付、Git提交或GitHub推送。
